1. 处理流程

Step 1：读取与分析

列出 10-inbox/ 下所有文件，分析每份文件的主题、核心观点、重复内容和关联关系。

Step 2：格式处理

2.1 YAML Frontmatter

每份文档必须添加/修复 YAML，各占独立行，禁止压缩成单行：

---
title: "文档标题"
tags:
  - 岗位/技术栈    # 从【岗位|考点|难度|题型】四维度提取
  - 考点/分类
  - 难度/级别
  - 题型/类型
  - tier/层级       # T1概览 T2锚点 T3参考
category: 主分类/子分类
create_date: 处理当天日期
source:           # 仅从inbox整理时保留
  - 原文件1.md
---

标签四维度：

  • 岗位：K8s / Linux / 网络 / MySQL / Redis / Kafka / ES / 容器 / Python / AI / 英语
  • 考点：原理 / 实操 / 故障排查 / 面试口述
  • 难度：入门 / 中级 / 高阶 / 大厂深挖
  • 题型：简答题 / 场景题 / 八股 / 手写代码

层级标签（如 岗位/K8s）中的 / 是有效的，不要误判为路径分隔。 tags 条目绝不能包含 .md 后缀（那是 source 字段的内容）。

2.2 标题序号

级别格式示例
H1（文档标题）不编号，唯一# Deployment 详解
H2数字. ## 1. , ## 2.
H3数字.数字. ### 1.1. , ### 2.1.
H4数字.数字.数字. #### 1.1.1.

规则：

  • 移除中文数字（一、二、三 → 1. 2. 3.）
  • emoji 保持在编号之前：📖 1. 标题
  • 同一文档禁止出现相同编号
  • 编号连续、无跳跃

2.3 代码块

  • 必须添加语言标记：yaml / bash / json / text
  • 闭合 fence 是裸 ```，不能带语言标记
  • 无语言标记时通过相邻行推断：
相邻行特征推断语言
apiVersion、kind: 开头yaml
kubectl、helm、docker、curl、yum、# 开头bash
{ 开头json
< 开头xml
其他text

2.4 表格与 Callout

表格规范： 列数 <= 4、表前后空行、对齐线对齐。

Callout 规范：

类型用途
> [!info]核心概念、定义
> [!tip]最佳实践
> [!warning]注意事项
> [!danger]高风险操作

图标规范（适度使用）：

  • ✅ 📖 章节 / 🔧 配置 / 💡 要点 / ⚠️ 警告 / ✅ 完成
  • ❌ 每行都加图标

Step 3：内容增强

3.1 内容深度要求

  • 不仅列要点，要讲清原理（为什么这样设计）
  • 技术概念说明底层机制
  • 故障场景包含排查思路和步骤

3.2 交叉链接

每文档末尾添加 > [!info] 关联文档，至少 3-5 个相关链接：

  • 同技术栈（Deployment ↔ ReplicaSet ↔ StatefulSet）
  • 上下游依赖（PV&PVC ↔ StorageClass ↔ 存储体系）
  • 对比补充（硬中断 ↔ 软中断 ↔ 内核学习）
  • 链接前验证目标文件存在

3.3 英语词汇提取

处理技术文档时提取英文术语到 20-areas/english/vocabulary/：

  • 来源：代码块内的术语、技术专有名词
  • 格式：表格（术语 / 中文释义 / 说明）
  • 已存在的术语不重复添加

3.4 面试题格式

严格按以下 4 模块：

【题目】：原样保留题干
【参考答案】：分点作答，原理+踩分点分开
【延伸追问】：2-3 条衍生问题
【易错坑点】：面试易错细节

Step 3.5：知识锚点去重

3.5.1 知识分级体系

每个文档在 YAML tags 中标记知识层级：

层级tags 标记定位示例
T1 概览tier/overview讲广度，给全貌，深度内容用链接跳转K8s 核心概念.md
T2 锚点tier/anchor讲透一个知识点，全局唯一深度入口pause 容器完整详解.md
T3 参考tier/reference侧重实操/故障，假设读者已懂原理K8s 故障排查手册.md

3.5.2 锚点文档的内容边界

T2 锚点文档在 YAML 中声明 scope 和 out_of_scope：

scope: "本知识点覆盖的范围"
out_of_scope: "明确不属于本知识点范围的内容"

超出 scope 的内容切分到对应领域的其他锚点文档，不留孤儿知识。

3.5.3 处理 inbox 时的去重决策

发现文件 A 和文件 B 都讲同一知识点
├── 检查该知识点是否已有 T2 锚点文档
│   ├── 有 → 将 A、B 的新信息合并入锚点，A、B 不新建
│   └── 无 → 创建新锚点，内容 = A+B 去重整合
│
└── 检查 A、B 中超出锚点 scope 的内容
    ├── 有 → 切分到对应领域锚点
    └── 无 → 完成

3.5.4 文档合并时的深度规则

合并文档 A + B（如两份都讲 Deployment）：
1. 保留 A 中讲得最好的原理部分
2. 保留 B 中 A 没有的补充内容
3. 删除 A 和 B 的重复部分（保留更完整的那份）
4. 如果 A 讲原理、B 讲实操 → 拆为两个文档：
   - Deployment 完整深度详解 (T2 锚点，原理)
   - Deployment 故障排查 (T3 参考，实操)

3.5.5 去重验证清单

  • 该知识点是否已有 T2 锚点？有则合并入锚点
  • 当前文件内容是否超出 scope？超出则拆分
  • 合并文档有无深度层级冲突？T1 不深入 T2 细节
  • 锚点索引是否更新？

Step 3.6：多文档内容重叠处理

3.6.1 重叠判定标准

重叠类型判定条件处理策略
重度重叠两文档 ≥60% 内容重复抽取共享知识为独立 T2 锚点，双端引用
背景引用A 以 B 的知识为前置条件非锚点文档中 ≤3 行概述 + 详见 [[锚点]]
视角互补同一概念不同维度（原理 vs 实操）各自保留，交叉链接，内容边界约定
多源汇聚同一知识点来自不同渠道（官方+博客+自总结）合并择优，用 > [!info] 标注来源
包含关系A 的主题是 B 的子集主体归锚点，子集只留概述 + 链接

3.6.2 非锚点文档的内容限制

  • 非 T2 锚点文档中，引用其他知识点的正文描述 ≤ 3 行
  • 超过 3 行 → 替换为 详见 [[锚点文档#章节名]]
  • T3 故障排查/面试题文档开头加前置说明： 
    > 本文假设读者已掌握 [知识点] 基本原理，详见 [[锚点文档]]

3.6.3 内容边界检查清单

  • 当前文档是否与已有文档存在内容重叠？
  • 重叠部分是否已由锚点文档完整覆盖？
  • 非锚点文档中对重叠知识的描述是否 ≤ 3 行？
  • 是否需要新建共享知识锚点？
  • 跨文档链接是否精准到具体章节？

3.6.4 跨文档链接的精准锚点语法

引用粒度语法示例
整篇引用详见 [[文档标题]]详见 [[pause 容器完整详解]]
章节引用详见 [[文档标题#章节号. 章节名]]详见 [[pause 容器完整详解#4. pause 故障影响]]
故障场景引用排查步骤详见 [[故障排查文档#场景X]]排查步骤详见 [[K8s 故障排查手册#场景3 Pod CrashLoopBackOff]]

Step 4：分类归档

4.1 分类原则

  • 单文件单主题 — 一个文档只讲一个知识点
  • 跨类别拆分 — 不同技术的合并文件拆为独立文件
  • 故障排查与理论分离 — 理论放根目录，故障放 故障排查/
  • 面试题统一放 面试题/ 子目录
  • 分类不满足时新建子目录

4.2 官方文档处理

文件名标记为「官方文档」的文档：按标准流程整理后放入对应技术目录，原文档放入 50-archives/ 对应分类。

4.3 已完成标记

文件名标记为「已完成」但未整理内容的文档：仅做打标签、分类、添加链接，不改变正文。

4.4 未完成文件

文件名标记为「未完成」的文档禁止任何操作，保留原位。

4.5 特殊文件

  • 收件箱.md — 保留不动
  • 图片/附件 → 建议移入 90-attachments/
  • 重复/无效素材 → 标记「可删除」并说明理由

Step 5：最终验证清单

代码块检查：

  • 所有代码块有语言标记
  • 闭合 fence 是裸 ```，无语言标记
  • 前后代码块无粘连

标题检查：

  • 无中文数字编号
  • 无重复序号
  • H4 仅出现在 H3 下方

YAML 检查：

  • 各字段独立成行
  • tags 中无 .md 结尾
  • create_date 为当天日期

内容检查：

  • 无连续 3 个以上空行
  • 表格前后有空行、列数一致
  • 关联文档使用 [[wikilink]]
  • Callout 使用标准格式

强制执行： 任一检查不通过 → 退回修改，不完成归档。

2. 技术规则附录

2.1 YAML tags 操作铁律

  1. 不要用含 / 的判断来分离 tags 和 source — 层级标签（岗位/K8s）是有效的
  2. 各字段的 - item 含义不同：tags=标签 / source=来源 / aliases=别名 / cssclasses=样式
  3. 修复 YAML 前先备份 — 一次错误 replace 可能让所有 tags 失效
  4. 修改后验证：
    • tags 中无 .md 结尾
    • tags 中无 wide-page（来自 cssclasses）
    • tags 中无从 aliases 泄漏的条目
    • source 字段存在时必须有 - filename.md

2.2 PowerShell 避坑

  1. 禁止 heredoc（<<） — 使用 @'...'@ 包裹字符串
  2. Python 脚本中的 $ — 用单引号字符串避免 PowerShell 变量展开
  3. 编码 — 始终 -Encoding UTF8
  4. 大文件写入 — 禁止字符串拼接，使用临时 .py 脚本文件写入

2.3 文档合并规则

  1. 禁止机械拼接 — 需总结、去重、按逻辑重组
  2. 相同主题取优 — 保留更好的一份，另一份引用
  3. 加过渡段落 — 不同来源之间加过渡文字
  4. 合并后验证：无重复段落、逻辑连贯、标题序号唯一

2.4 文档重写条件

仅在格式严重错乱时重写（代码块断裂、标题丢失、内容错位），保留原始技术信息，不添加 AI 臆造内容。

2.5 链接安全规范

  1. 使用 [[wikilink]]，不用绝对路径
  2. 不链接不存在的文件
  3. 文件夹链接禁止尾部斜杠
  4. 文件名含 & = + 时避免歧义
